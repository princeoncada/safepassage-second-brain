# Schemas and Metadata Standards

## Purpose

This file defines the required metadata, document types, ingestion payloads, folder routing rules, and naming standards for the SafePassage Second Brain system.

Markdown files inside `vault/` are the canonical source of truth.

DOCX and PDF files are export artifacts only.

---

## Required Markdown Metadata

Every important document must start with this YAML block:

```yaml
---
title:
type:
community:
priority:
effective_date:
source:
status:
tags:
last_updated:
version:
---
```

---

## Initial Ingestion Payload

This is the standard JSON shape n8n and the AI classifier will use.

```json
{
  "input_text": "",
  "source": "",
  "community": "",
  "received_at": "",
  "classification": "",
  "priority": "",
  "target_folder": "",
  "suggested_filename": "",
  "normalized_markdown": ""
}
```

---

## Ingestion Payload Field Definitions

| Field | Description |
|---|---|
| input_text | Raw user-provided information |
| source | Where the information came from |
| community | Community name, or `global` |
| received_at | Timestamp of ingestion |
| classification | AI-classified document type |
| priority | low, medium, high, critical |
| target_folder | Vault folder where the file should be saved |
| suggested_filename | Safe filename generated by AI or n8n |
| normalized_markdown | Final Markdown content to write into the vault |

## Filename Convention

Use lowercase kebab-case.

Format:

```txt
yyyy-mm-dd-community-topic-type.md
```

Example:

```txt
2026-05-14-sierra-ridge-physical-id-rule-post-order.md
```

For global documents:

```txt
2026-05-14-global-visitor-verification-qa-rule.md
```

---

## Versioning Rules

Use this format:

```txt
1.0 initial version
1.1 minor wording update
1.2 added clarification
2.0 major rule change
```

Never overwrite important rule changes without updating:

```txt
version
last_updated
Change History
```

---

## Tags Standard

Tags should be lowercase kebab-case.

Good:

```yaml
tags:
  - visitor-access
  - id-verification
  - qa-risk
```

Bad:

```yaml
tags:
  - Visitor Access
  - ID stuff
  - important!!!
```

---

## Normalized Markdown Requirements

Every normalized document must be:

- human-readable
- structured
- metadata-complete
- short enough for retrieval chunks
- clear enough for a human without AI
- specific enough for QA protection

Do not save vague notes without context.

---

## Source of Truth Rule

Markdown in `vault/` is the source of truth.

These are not source of truth:

```txt
exports/docx/
exports/pdf/
n8n executions
Open WebUI chats
AI responses
```

If information matters long-term, it must become a Markdown file in the vault.

# SafePassage Primary Kiosk Workflow Sample Input

This sample is manually structured from the base kiosk training workflow. It is default guidance only. Always check community post orders and announcements for overrides.

## primary-kiosk-basics

Source Section: Kiosk Basics
Source Page: unknown

Everyone needs to show ID. No ID means no entry.

Default exceptions include pre-approved marked vehicles, residents in the car, emergency vehicles, and registered tags.

Always check community post orders and announcements for community-specific overrides.

## primary-call-attempts-by-community

Source Section: Resident Call Attempts
Source Page: unknown

Default workflow is to call the resident twice.

Some communities differ. Always check the community post order for accuracy before treating the default call-attempt count as final.

## primary-saving-tags-procedure

Source Section: Saving Tags
Source Page: unknown

Use the saving tags script when a tag needs to be saved.

Always check the community post order for accuracy before saving or changing tag handling.

Source Section: Before Shift Screenshots

## primary-community-client-codes

Source Section: Community Client Codes
Source Page: unknown

Use the correct community and client code when processing kiosk interactions.

If the correct code is unclear, stop and check the current community reference before proceeding.

## primary-kiosk-call-flow

Source Section: Kiosk Call Flow
Source Page: unknown

Default kiosk call flow:

1. Greet the guest or vendor.
2. Ask for the address.
3. Ask who they are visiting or which company they are with.
4. Ask for company spelling when needed.
5. Request physical ID.
6. Ask for the resident name.
7. Call the resident if needed.
8. Add tile or access when approved.
9. Save tag if needed.
10. Log the entry.
11. Open the gate.

Always check post orders and announcements for community-specific changes.

## primary-calling-resident-for-access

Source Section: Calling Resident
Source Page: unknown

Use the recorded line script when calling the resident.

Ask whether the resident is expecting the visitor or vendor.

Ask how many access days should be added when access is approved.

Add the visitor or vendor name and approved access details.

Deny entry if the resident declines access.

## primary-leaving-voicemail

Source Section: Voicemail
Source Page: unknown

Leave a voicemail if the resident does not answer.

Follow the current voicemail script and keep the message concise and professional.

## primary-deny-entry

Source Section: Deny Entry
Source Page: unknown

Use the deny-entry script when the resident does not answer or when the resident is not expecting the visitor or vendor.

Explain the denial clearly and professionally.

Do not open the gate when access is not approved.

## primary-no-physical-id

Source Section: No Physical ID
Source Page: unknown

Default workflow says to deny entry when the guest or vendor has no physical ID.

Use the no physical ID reason when recording the denial.

Always check post orders and announcements for community-specific ID rules or approved exceptions.

## primary-incomplete-interaction

Source Section: Incomplete Interaction
Source Page: unknown

If a guest or vendor leaves after the interaction begins, notify the resident when appropriate.

Do not process a log entry or deny entry when the guest or vendor leaves before completion.

End the call in the kiosk timer.

## primary-spanish-translation

Source Section: Spanish Translation
Source Page: unknown

Use the approved Spanish translation guidance when needed.

Keep the interaction professional and follow the same verification and access-control steps.

## primary-ticket-format

Source Section: Ticket Formats
Source Page: unknown

Use the correct ticket format for the issue type.

Supported ticket examples include tailgating, gate issue, residency confirmation, and tag issue.

Include the relevant community, lane or gate if known, vehicle or tag details if available, and the action taken.
